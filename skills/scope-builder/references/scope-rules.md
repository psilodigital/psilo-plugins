# Scope Rules

How Psilodigital scopes software work.

## Always

- Start from the business outcome, not the feature list.
- Make MVP smaller than the operator wants.
- Phase work so each phase ships something usable.
- List capabilities by what the user does, not by services or files.
- Be explicit about what is NOT in scope.
- Surface assumptions in their own section.
- Flag every integration as live / planned / unknown.

## Never

- Assume an API exists without checking documentation.
- Promise a specific tool integration without confirming auth model and rate limits.
- Bundle infrastructure into product capabilities without showing it.
- Forget to ask who owns credentials.
- Quote effort as a single point. Always use a range.

## Sizing language

- S = 1–3 days
- M = 4–10 days
- L = 11–25 days
- XL = needs further breakdown — flag it, don't accept it

## When to push back

- Scope items that are actually V2.
- "Just a small AI feature" without a measurable use case.
- Replacing tools that already work because they offend us aesthetically.
- Vague capabilities like "make it fast" — define fast.

## Change requests

Anything not in the signed scope is a CR. CRs are sized S/M/L and priced separately.
